# Debug Notes

把「剛剛這個對話裡踩到、又解掉的坑」寫進專案根目錄的 [DEBUG_NOTES.md](../../../DEBUG_NOTES.md)，格式跟既有內容一致，讓下一個人（或下一次的自己）不用重踩一次。

## 先讀 DEBUG_NOTES.md，不要憑空套格式

打開檔案，看：
- 目前最新的章節編號（`## N.`），新條目接續編號，不要重新從 1 開始 — 除非是全新主題（見下面「新主題要不要開新 section」）。
- 既有條目的語氣和結構：**現象 → 原因 → 修法 → 教訓**（不是每條都四個標籤都出現，但大致是這個順序），全部用繁體中文，技術細節要具體到能重現/驗證，不要寫模糊的「要注意 XXX」。
- 原因部分優先寫**驗證方式**，不是只寫推測 — 既有條目常見「實測驗證方式」這個標籤，說明是怎麼用數據/對照組確認因果關係的，不是純靠猜。

## 什麼值得寫、什麼不值得

**值得寫**：
- 花了不只一輪來回才抓到根因的 bug，尤其是「照文件/直覺應該要動,但實際上沒動」這種跨系統邊界的行為差異。
- 中途繞了彎路的（例如先猜了一個錯的原因、修了沒用，才找到真正原因）——彎路本身是有價值的資訊，讓下一次能更快排除掉錯的假設。
- 最後採用的修法背後有非顯而易見的取捨（例如「為什麼不能直接 pause() 影片」「為什麼疊圖蓋不住」這種一般人直覺會走錯路的地方）。

**不值得寫**：
- 一眼就能看出原因的小 typo / 邏輯錯誤。
- 已經在程式碼註解裡講清楚的東西——DEBUG_NOTES.md 是給「還沒讀到那段程式碼之前就該知道」的坑，不是重複程式碼裡已經有的說明。
- 例行的功能開發過程，沒有真正「卡住、繞彎路、最後才搞懂」的部分。

拿不準時，優先問自己：如果沒寫這篇，下一次遇到同類狀況會不會浪費一輪以上的來回才重新搞懂？不會的話就不用寫。

## 新主題要不要開新 section

DEBUG_NOTES.md 目前的章節編號是全檔案共用的、遞增的，但每個大主題（目前是「開發 nestedScene3PopupSpawner.js」跟「eye.eyeLayer.js 的 `<video>` 眨眼播放坑」）各自有自己的 `---` 分隔線 + 一段簡短的情境介紹（在講哪個檔案/哪個功能），再接著它自己的 `## N.` 條目列表，最後可能有一段「共通教訓」總結。

- 如果新條目跟檔案裡**最後一個既有主題**是同一個檔案/同一個功能範圍 → 直接在該主題底下加一條新的 `## N.`（編號接續全域最大值），不用開新 section。
- 如果是**全新的檔案/功能** → 在檔案最後加 `---` 分隔線 + 一段一兩句話的情境介紹（模仿既有 section 開頭的寫法：「開發/除錯 `<檔案路徑>`（<這個檔案是做什麼的>）過程中踩到的坑」），再接新的 `## N.` 條目，編號一樣接續全域最大值（不要重新從 1 開始 — 現有第二個主題是從 `## 8.` 接續第一個主題的 `## 1.`-`## 7.`）。

## 寫作檢查清單

- [ ] 中文（繁體），跟既有條目語氣一致，不要混英文技術詞彙以外的內容進中文句子裡breaking 語感。
- [ ] 標題（`## N. 一句話描述症狀或修法`）本身就要能讓人一眼判斷「這跟我現在卡住的問題像不像」。
- [ ] 「現象」段落寫具體症狀，不是抽象描述——最好是「做了 X，預期 Y，結果看到 Z」的形式。
- [ ] 「原因」段落如果有做過實測驗證（比對數值、截圖、log），寫出驗證方式和結果，不是只寫結論。
- [ ] 「修法」段落如果最後採用的方案跟第一次嘗試的不一樣，兩個都寫——包含「為什麼第一個方案不夠」。
- [ ] 具體的檔名/函式名用 `code` 格式標出來，方便之後用 grep 找到對應程式碼。
- [ ] 寫完後不要自己再摘要一次「以上就是...」，條目本身結束就結束。

## 完成後

只要用 Edit/Write 把新內容加進 DEBUG_NOTES.md 就好，不用額外建立其他檔案、不用另外通知使用者細節（除非使用者本來就在等你回報）——這是一個記錄動作,不是需要額外討論的變更。
